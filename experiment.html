SQL Fundamentals

Author: Roland Tuboly

Part I. Foundational Query Structure

I.1. Query Execution and Basic Clauses

Learning goals: Understand SQL execution phases and basic data retrieval filtering.

Prerequisites: Basic understanding of tabular data.

Key terms: Clause, Query, Execution Order, Filter.

I.1.1. Data Definition and Manipulation

Data Definition Language (DDL) includes CREATE, ALTER, and DROP for managing databases, tables, and indexes. Data Manipulation Language (DML) modifies data states: INSERT INTO adds new records, UPDATE modifies existing records, and DELETE removes records. Database systems may utilize TOP or FETCH FIRST n ROWS ONLY as syntactic alternatives to LIMIT.

I.1.2. Execution Order vs. Syntax Order

The written syntax order of a query is SELECT - FROM - WHERE - GROUP BY - HAVING - ORDER BY - LIMIT. A common logical processing order is FROM - WHERE - GROUP BY - HAVING - SELECT - DISTINCT - ORDER BY - LIMIT, though the optimizer may choose a different physical execution plan.

I.1.3. SELECT, FROM, and DISTINCT

SELECT dictates columns to return, while DISTINCT forces unique values and can combine multiple columns. Views can be stored for reuse using CREATE VIEW ... AS, which defines a reusable query object rather than returning a result set immediately. LIMIT (number) restricts output to the first specified number of observations.

SELECT DISTINCT continent, country 
FROM nations 
LIMIT 5;
Output: Returns up to 5 unique continent-country pairs.

I.1.4. Filtering with WHERE

WHERE filters records before aggregation. Operators include <> (not equal), OR, AND, BETWEEN ... AND (inclusive), IN () for multiple values, and IS NULL / IS NOT NULL.

SELECT title 
FROM films 
WHERE certification IN ('G', 'PG') 
AND release_year > 2010;
Output: Returns titles of films with 'G' or 'PG' certifications released after 2010.

I.1.5. Grouping and Filtering Aggregations

GROUP BY groups results, usually combined with an aggregate function. HAVING filters grouped records after aggregation.

SELECT certification, COUNT(title) AS title_count 
FROM films 
GROUP BY certification 
HAVING COUNT(title) > 500;
Output: Returns certifications and their counts, only for groups exceeding 500.

I.1.6. Sorting

ORDER BY sorts the result set (ASC or DESC) and can evaluate multiple fields sequentially.

Common mistakes: Using WHERE instead of HAVING to filter aggregated metrics.

Quick recap: WHERE filters rows, GROUP BY aggregates rows, HAVING filters aggregations.

Practice question: How do you filter a dataset to show only departments with more than 10 employees?

Answer: GROUP BY department HAVING COUNT(employee_id) > 10;

I.2. Aggregation and Arithmetics

Learning goals: Compute numerical statistics and perform column math.

Prerequisites: Section I.1.

Key terms: Aggregation, Arithmetic Operators, Rounding.

I.2.1. Basic Aggregation Functions

Aggregate functions summarize groups of rows into single values. They are often aliased for readability, though aliases are optional.

  • COUNT(...): Counts records with non-null values in a field.
  • COUNT(*): Counts all records, including nulls.
  • COUNT(DISTINCT ...): Counts unique values.
  • AVG(), SUM(), MAX(), MIN(): Standard mathematical aggregations. COUNT, MIN, and MAX work on non-numerical data.
SELECT 
    COUNT(*) AS total_rows, 
    COUNT(DISTINCT category) AS unique_categories,
    AVG(price) AS average_price
FROM inventory;
Output: Returns the total count, number of unique categories, and average price.

I.2.2. Mathematical Operators and Rounding

Arithmetic operations include + - * /. Integer division requires decimals (e.g., 3.0/2.0) to avoid truncation. ROUND() accepts negative parameters to round to the nearest 10, 100, etc., and omitting the second parameter rounds to a whole number.

SELECT ROUND(AVG(price), 2) AS avg_price 
FROM products;
Output: The average price rounded to two decimal places.

Common mistakes: Dividing two integers and expecting a float result.

Quick recap: Use explicit float values for exact division; aggregate functions ignore nulls except COUNT(*).

Practice question: How do you find the total number of unique countries in a table?

Answer: SELECT COUNT(DISTINCT country) FROM table_name;

Part II. Data Types and Manipulation

II.1. Data Types

Learning goals: Identify and query standard database column types.

Prerequisites: None.

Key terms: VARCHAR, TIMESTAMP, INTERVAL, ARRAY, ENUM.

II.1.1. Text, Numeric, and Array Data Types

Text types include CHAR, VARCHAR, and TEXT. Numeric types include INT and DECIMAL. Arrays allow storing lists (e.g., int[] or text[][]). PostgreSQL array indexes start at 1. Arrays can be searched using ANY() or the @> match operator.

SELECT name, scores 
FROM students 
WHERE 90 = ANY(scores);
Output: Returns names and score arrays for students who have at least one score of 90.

II.1.2. Date, Time, and Timestamp Types

Dates use ISO 8601 (yyyy-mm-dd). TIMESTAMP includes date and time. TIME stores time without timezone by default. INTERVAL stores time durations.

II.1.3. User-Defined Types

Enumerated types (ENUM) define static, custom categorical lists.

CREATE TYPE dayofweek AS ENUM ('Monday', 'Tuesday');
Output: Creates a custom type restricting column values to the defined days.

II.1.4. Querying Information Schema

Extract table metadata using INFORMATION_SCHEMA.COLUMNS or pg_type.

SELECT column_name, data_type 
FROM INFORMATION_SCHEMA.COLUMNS 
WHERE table_name = 'film';
Output: Returns column names and their corresponding data types.

II.1.5. JSON and Semi-Structured Data

PostgreSQL's JSONB type stores binary-encoded JSON, supporting indexing and faster processing. Extraction operators: -> (returns JSON object) and ->> (returns text). Search using the containment operator @>.

SELECT metadata->>'browser' AS browser
FROM web_logs
WHERE metadata @> '{"os": "Linux"}';
Output: Extracts the 'browser' field from logs where the 'os' is 'Linux'.

Common mistakes: Zero-indexing PostgreSQL arrays, using -> when text output (->>) is required for comparison.

Quick recap: Arrays use 1-based indexing; JSONB is the preferred format for semi-structured data; metadata is accessible via INFORMATION_SCHEMA.

Practice question: How do you extract a nested 'city' value from a JSONB column 'address' as text?

Answer: address->'location'->>'city'

Practice question: How do you query the first element of an array column named scores?

Answer: SELECT scores[1] FROM table_name;

II.2. String and Character Manipulation

Learning goals: Modify, parse, and format string data.

Prerequisites: Section II.1.

Key terms: Concatenation, Substring, Padding.

II.2.1. Concatenation and Case Modification

Concatenate strings using || or CONCAT(). Cast cases using UPPER(), LOWER(), or INITCAP() (title case).

II.2.2. Parsing and Substrings

Extract segments using LEFT(string, length) or RIGHT(string, length). Find index positions using POSITION('char' IN string) or STRPOS(string, 'char'). SUBSTRING(source, start, length) extracts characters. Length is checked via CHAR_LENGTH() or LENGTH().

II.2.3. Trimming, Padding, and Replacing

TRIM() removes leading, trailing, or both characters (defaults to blank space). LPAD() and RPAD() pad strings with specific characters to reach a defined length, truncating if the target length is smaller. REPLACE() swaps substrings. REVERSE() reverses string characters.

SELECT LPAD('padded', 10, '#');
Output: ####padded.

Common mistakes: Confusing POSITION output with 0-indexed languages.

Quick recap: String functions allow extraction, padding, and case manipulation directly in the SELECT clause.

Practice question: Extract the first 3 characters of a column code.

Answer: SELECT LEFT(code, 3) FROM table_name;

II.3. Time Data Manipulation

Learning goals: Perform arithmetic and extraction on timestamps.

Prerequisites: Section II.1.

Key terms: INTERVAL, EXTRACT, DATE_TRUNC.

II.3.1. Date and Time Arithmetic

Subtracting dates yields an integer representing days. Adding an integer to a date defaults to adding days. Subtracting timestamps yields an INTERVAL. AGE() calculates periods between timestamps. INTERVAL types can be mathematically manipulated.

SELECT 
    rental_date + INTERVAL '3 days' AS due_date,
    AGE(return_date, rental_date) AS rental_duration
FROM rentals;
Output: Returns the due date (3 days later) and the total duration of the rental.

II.3.2. Extracting and Truncating Time Data

EXTRACT(field FROM source) and DATE_PART('field', source) retrieve sub-fields like quarter, year, or dow (day of week). DATE_TRUNC('field', timestamp) rounds down a timestamp to the specified precision.

SELECT DATE_TRUNC('month', TIMESTAMP '2005-05-21 15:30:30');
Output: 2005-05-01 00:00:00.

II.3.3. Current Date and Time Functions

NOW() returns a timestamp with microsecond precision and timezone. CURRENT_TIMESTAMP(precision) acts similarly. CURRENT_DATE and CURRENT_TIME provide lower precision alternatives. Cast without timezone using NOW()::timestamp.

Common mistakes: Using standard subtraction on timestamps and expecting an integer.

Quick recap: EXTRACT gets a number out of a date; DATE_TRUNC resets a date to the start of a specific period.

Practice question: Get the current year as an integer.

Answer: SELECT EXTRACT(year FROM CURRENT_DATE);

II.4. Advanced Searching and Extensions

Learning goals: Perform fuzzy and full-text searches.

Prerequisites: Section II.2.

Key terms: tsvector, pg_trgm, Levenshtein.

II.4.1. Pattern Matching

LIKE is case-sensitive, using % for zero/multiple characters and _ for exactly one character. NOT LIKE reverses this.

II.4.2. Full-Text Search

Converts text into a tsvector (a sorted list of normalized word variants called lexemes). to_tsvector and to_tsquery are compared using the @@ match operator, which is case-insensitive.

SELECT title 
FROM film 
WHERE to_tsvector(title) @@ to_tsquery('elf');
Output: Titles containing the word "elf" regardless of casing.

II.4.3. PostgreSQL Extensions

Extensions like PostGIS, PostPic, fuzzystrmatch, and pg_trgm expand capabilities.

  • Levenshtein(str1, str2) calculates the edit distance between strings.
  • similarity(str1, str2) uses trigrams (3 consecutive characters) to score similarity from 0 to 1.

Common mistakes: Using LIKE for complex text searches instead of full-text vectors.

Quick recap: LIKE is basic pattern matching; tsvector enables fast, robust textual search.

Practice question: How do you query installed extensions?

Answer: SELECT extname FROM pg_extension;

Part III. Combining and Structuring Data

III.1. Joins

Learning goals: Merge multiple tables horizontally.

Prerequisites: Section I.1.

Key terms: INNER, OUTER, CROSS, USING.

III.1.1. Standard Joins

  • INNER JOIN: Returns records matching the condition in both tables. USING(column_name) replaces ON if fields are identical.
  • LEFT (OUTER) JOIN: Returns all left table records and matching right table records; missing matches are null.
  • RIGHT (OUTER) JOIN: Reverses LEFT JOIN logic.
  • FULL (OUTER) JOIN: Combines LEFT and RIGHT joins.
SELECT p1.country, p2.continent 
FROM presidents AS p1 
INNER JOIN prime_ministers AS p2 
ON p1.country = p2.country;
Output: Data for countries possessing both a president and prime minister.

III.1.2. CROSS JOIN and Self Joins

CROSS JOIN creates all possible combinations between two tables without an ON or USING clause. Self joins connect a table to itself, requiring aliases.

III.1.3. Semi and Anti Joins

  • Semi join: Filters left table records when a match exists in the right table, commonly via WHERE EXISTS (...) or WHERE ... IN (subquery).
  • Anti join: Filters left table records when no match exists in the right table, typically via WHERE NOT EXISTS (...) or LEFT JOIN ... WHERE right_table.id IS NULL.

III.1.4. Chaining Multiple Joins

Multiple joins sequentially link tables, potentially utilizing AND within the ON clause for multiple conditions.

Common mistakes: Forgetting aliases in a self join.

Quick recap: Joins scale columns horizontally; semi/anti joins strictly act as filters without adding columns.

Practice question: Create a Cartesian product of table A and B.

Answer: SELECT * FROM A CROSS JOIN B;

III.2. Set Theory

Learning goals: Merge result sets vertically.

Prerequisites: Section I.1.

Key terms: UNION, INTERSECT, EXCEPT.

III.2.1. UNION and UNION ALL

Stacks fields vertically; requires the same number of columns and data types. UNION returns unique records; UNION ALL includes duplicates. The resulting column names are inherited from the left query. Requires no join field.

SELECT name FROM customers
UNION
SELECT name FROM suppliers;
Output: A combined list of unique names from both customers and suppliers.

III.2.2. INTERSECT and EXCEPT

  • INTERSECT: Returns records existing in both result sets, omitting duplicates.
  • EXCEPT: Retains only left table records absent from the right table. The entire record must match.

Common mistakes: Trying to UNION tables with mismatched column structures.

Quick recap: Joins combine horizontally; Set operators stack vertically.

Practice question: How do you stack query results while keeping duplicates?

Answer: UNION ALL.

III.3. Conditional Logic

Learning goals: Create derived columns using logic statements.

Prerequisites: Section I.1.

Key terms: CASE, WHEN, THEN, ELSE.

III.3.1. CASE WHEN, THEN, ELSE, END

Evaluates to one new column. Requires END and usually an alias. If ELSE is omitted, unmatched conditions return null. Multiple conditions can use AND inside the WHEN clause.

SELECT CASE 
    WHEN x = 1 THEN 'a' 
    WHEN x = 2 THEN 'b' 
    ELSE 'C' 
END AS new_column;
Output: Generates a column mapping 1 to 'a', 2 to 'b', and all others to 'C'.

III.3.2. Categorization, Filtering, and Aggregating

CASE statements inside aggregate functions selectively filter data for counts or sums.

SELECT season, COUNT(CASE WHEN home_goal > away_goal THEN 1 END) AS home_wins 
FROM match 
GROUP BY season;
Output: Returns the number of home wins per season.

Common mistakes: Forgetting the END keyword.

Quick recap: CASE functions as an IF-THEN-ELSE logical structure directly inside SELECT or aggregation blocks.

Practice question: Assign 'Adult' if age >= 18, else 'Minor'.

Answer: CASE WHEN age >= 18 THEN 'Adult' ELSE 'Minor' END

III.4. Subqueries and Transformations

Learning goals: Nest queries for complex filtering and structure.

Prerequisites: Section I.1., III.1.

Key terms: Subquery, Correlated, CTE.

III.4.1. The EXISTS Operator

The EXISTS operator evaluates whether a subquery returns any records, acting as a boolean filter.

III.4.2. Subqueries in WHERE, FROM, and SELECT

  • WHERE: Returns scalars, lists, or tables for filtering.
  • FROM: Restructures data and allows aggregations of aggregations. Requires an alias.
  • SELECT: Must return a single value. Processes before main query.

III.4.3. Correlated vs. Nested Subqueries

  • Nested: Uncorrelated subqueries execute independently.
  • Correlated: Reference values from the outer query. Re-runs for every row generated by the main query, resulting in high processing times.

III.4.4. Common Table Expressions (WITH)

CTEs organize queries sequentially. They can reference earlier CTEs, and recursive CTEs can reference themselves. They are defined with the WITH clause and may be materialized or inlined depending on the SQL engine and query plan.

WITH s1 AS (
    SELECT id FROM match WHERE home_goal >= 10
) 
SELECT COUNT(id) FROM s1;
Output: Defines a temporary table s1 holding matches with 10+ home goals, then counts them.

III.4.5. Strategic Selection

  • Joins: Simple combinations/aggregations.
  • Correlated Subqueries: Match subqueries to tables while avoiding join limits.
  • Nested Subqueries: Multi-step mathematical transformations.
  • CTEs: Readability and sequential logic.

Common mistakes: Returning multiple rows in a SELECT subquery.

Quick recap: Subqueries run internally to supply values; correlated subqueries depend on the outer query row-by-row; CTEs define clean, reusable temp results.

Practice question: Which technique is best for creating a readable multi-step query?

Answer: CTEs.

Part IV. Window Functions and Advanced Aggregation

IV.1. Window Function Fundamentals

Learning goals: Perform operations across related rows without collapsing them.

Prerequisites: Section I.2., III.4.

Key terms: OVER, PARTITION BY, Fetch, Rank.

IV.1.1. Anatomy of a Window Function

Window functions compute over a set of rows while keeping all rows in the output. Processed after standard queries except ORDER BY. Syntax: FUNCTION_NAME() OVER (PARTITION BY ... ORDER BY ...).

  • PARTITION BY: Resets the function operation across unique column values.
  • ORDER BY: Sorts the window locally.
SELECT 
    name, department, salary,
    AVG(salary) OVER(PARTITION BY department) AS dept_avg
FROM employees;
Output: Shows each employee alongside the average salary of their specific department.

IV.1.2. Fetching Values

  • LAG(column, n) / LEAD(column, n): Returns value n rows before/after current row.
  • FIRST_VALUE(column) / LAST_VALUE(column): Returns first/last value in partition. By default, the window ends at the current row.

IV.1.3. Ranking and Paging

  • ROW_NUMBER(): Assigns unique integers sequentially.
  • RANK(): Assigns identical numbers to ties, skipping subsequent ranks (e.g., 1, 2, 2, 4).
  • DENSE_RANK(): Assigns identical numbers to ties without skipping (e.g., 1, 2, 2, 3).
  • NTILE(n): Splits data into n approximately equal chunks (paging).
SELECT 
    title, score,
    DENSE_RANK() OVER(ORDER BY score DESC) AS rank
FROM games;
Output: Ranks games by score, with tied scores receiving the same rank and no gaps in the sequence.

Common mistakes: Forgetting that LAST_VALUE ends at the current row unless bounded.

Quick recap: Window functions analyze a group of rows related to the current row without grouping the final output.

Practice question: How to assign a strictly unique row ID regardless of ties?

Answer: ROW_NUMBER().

IV.2. Sliding Windows

Learning goals: Define bounds for running totals and moving metrics.

Prerequisites: Section IV.1.

Key terms: ROWS BETWEEN, RANGE BETWEEN, PRECEDING.

IV.2.1. Defining Frames

Defines calculations relative to current row within OVER().

  • ROWS BETWEEN: Evaluates strictly by physical row placement.
  • RANGE BETWEEN: Treats tied values in the ORDER BY clause as a single logical entity.

IV.2.2. Frame Keywords

PRECEDING, FOLLOWING, UNBOUNDED PRECEDING, UNBOUNDED FOLLOWING, CURRENT ROW.

SELECT 
    SUM(home_goal) OVER(
        ORDER BY date 
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM football_matches;
Output: Computes a cumulative sum from the first row to the current.

Common mistakes: Confusing ROWS and RANGE when handling duplicate sorting keys.

Quick recap: Frames restrict window operations to specific subsets relative to the current row.

Practice question: Frame declaration for the entire partition?

Answer: ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING.

IV.3. Advanced Grouping and Pivoting

Learning goals: Generate multi-dimensional rollups and pivot tables.

Prerequisites: Section I.2., IV.1.

Key terms: ROLLUP, CUBE, CROSSTAB, COALESCE.

IV.3.1. Hierarchical Aggregation

ROLLUP is a GROUP BY subclause generating hierarchical extra rows for group-level aggregations. ROLLUP(Country, Medal) yields Country-level totals and a grand total.

SELECT country, city, SUM(sales)
FROM global_sales
GROUP BY ROLLUP(country, city);
Output: Returns sales by city, country totals, and a grand total.

IV.3.2. Non-Hierarchical Aggregation

CUBE generates all possible group-level aggregations. CUBE(Country, Medal) creates Country-level, Medal-level, and grand totals.

IV.3.3. Handling Nulls

COALESCE() takes a list and returns the first non-null value from left to right. Essential for replacing nulls generated by ROLLUP or outer joins.

IV.3.4. Alternative Null Handling

IFNULL and ISNULL function similarly to COALESCE to replace null values, depending on the specific SQL dialect.

IV.3.5. Pivoting Data

The tablefunc extension provides CROSSTAB, moving row data into columns.

SELECT * FROM CROSSTAB($$source_sql$$) 
AS ct(col1 type1, col2 type2);
Output: Restructures vertical data horizontally.

IV.3.6. Data Compression

STRING_AGG(column_name, 'separator') aggregates string values into a single delimited string per group.

Common mistakes: Using CUBE on continuous dimensions causing combinatorial explosions.

Quick recap: ROLLUP totals chronologically/hierarchically; CUBE totals all dimensions.

Practice question: Replace nulls in commission with 0.

Answer: COALESCE(commission, 0);

Part V. Database Design and Storage Architecture

V.1. Data Structures and DBMS Types

Learning goals: Understand data formatting paradigms and their respective management systems.

Key terms: Structured, Unstructured, Semi-structured, RDBMS, NoSQL.

V.1.1. Data Formats

Structured data adheres to a rigid schema, such as relational database tables. Semi-structured data utilizes self-describing structures, such as JSON or XML. Unstructured data operates without a larger schema and constitutes the majority of global data.

V.1.2. Database Management Systems (DBMS)

SQL Database Management Systems (RDBMS) process structured, consistent data. NoSQL Database Management Systems manage rapid growth and schema-less data. They scale via document stores, key-value stores, columnar databases, and graph databases.

V.2. OLTP vs. OLAP

Learning goals: Differentiate between transactional and analytical processing architectures.

Key terms: OLTP, OLAP, Data Warehouse, Data Lake.

V.2.1. Processing Paradigms

Online Transaction Processing (OLTP) systems execute frequent updates to support daily transactions, utilizing application-oriented, highly normalized designs. Online Analytical Processing (OLAP) systems execute complex aggregations on historical data, utilizing subject-oriented, denormalized designs.

V.2.2. Storage Architectures

Data Warehouses store archived, structured data optimized for OLAP reading. Data Lakes store petabytes of structured and unstructured data using schema-on-read methodology. Data integration consolidates diverse source formats into a unified model through ETL (Extract, Transform, Load) or ELT pipelines.

V.3. Normalization

Learning goals: Apply normalization rules to eliminate data anomalies.

Key terms: 1NF, 2NF, 3NF, Anomaly.

V.3.1. Normal Forms

Normalization fragments tables to reduce data redundancy and increase integrity.

  • First Normal Form (1NF): Mandates strictly unique records and a single value per cell.
  • Second Normal Form (2NF): Requires 1NF compliance and dictates that non-key columns depend entirely on all parts of a composite primary key.
  • Third Normal Form (3NF): Requires 2NF compliance and eliminates transitive dependencies among non-key columns.

V.3.2. Anomalies

Failing to normalize generates update, insertion, and deletion anomalies.

V.4. Dimensional Modeling

Learning goals: Design schemas optimized for analytical querying.

Key terms: Fact Table, Dimension Table, Star Schema, Snowflake Schema.

V.4.1. Core Components

Dimensional modeling adapts relational frameworks for data warehouse OLAP queries. Fact tables record frequently changing metric events and connect to dimensions via foreign keys. Dimension tables store static or slowly changing descriptive attributes.

V.4.2. Schema Topologies

The star schema organizes one denormalized dimension table directly to a central fact table. The snowflake schema normalizes the dimension tables into sub-dimensions.

Part VI. Database Administration

VI.1. Materialized Views

Learning goals: Optimize read speeds for complex queries.

Key terms: Materialized View, Rematerialization.

Materialized views physically write query results to storage rather than executing the query upon each access. They demand scheduled or manual rematerialization to reflect underlying data changes. They optimize read-intensive OLAP environments and long-running queries.

VI.2. Roles and Access Control

Learning goals: Manage database security and user permissions.

Key terms: Role, GRANT, REVOKE.

Database roles function as users or groups to manage access permissions across a cluster. Specific privileges are assigned using the GRANT command and removed using the REVOKE command.

VI.3. Partitioning

Learning goals: Divide large tables to improve maintainability and query performance.

Prerequisites: Section III.4., VI.1.

Key terms: Vertical Partitioning, Horizontal Partitioning.

Vertical partitioning splits a table's columns to group frequently accessed data. Horizontal partitioning divides rows across multiple tables by specified ranges or lists. Partitioning ensures indices for heavy-use segments fit in memory, though retrofitting existing tables requires high operational overhead.

Part VII. Data Science and Analytics

VII.1. SQL for Feature Engineering

Learning goals: Create model-ready features and perform statistical analysis.

Prerequisites: Section I.2., IV.1., IV.2.

Key terms: Z-Score, Binning, One-Hot Encoding, Feature Scaling.

VII.1.1. Statistical Descriptive Features

Beyond basic averages, Data Science requires measures of dispersion. STDDEV() and VARIANCE() provide insight into data spread. Percentiles help identify outliers or describe distributions.

SELECT 
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY income) AS median_income,
    STDDEV(income) AS income_stddev
FROM users;
Output: Returns the median and standard deviation of the income column.

VII.1.2. Binning and Discretization

Transforming continuous variables into categorical "bins" is essential for certain models. This is typically achieved via CASE statements (manual bins) or NTILE() (equal-frequency bins).

VII.1.3. One-Hot Encoding

Categorical variables must often be converted into binary flags. This is performed using CASE statements inside aggregations to create a "pivot" effect.

SELECT 
    user_id,
    MAX(CASE WHEN color = 'Red' THEN 1 ELSE 0 END) AS is_red,
    MAX(CASE WHEN color = 'Blue' THEN 1 ELSE 0 END) AS is_blue
FROM preferences
GROUP BY user_id;
Output: A table with binary columns for each color category.

VII.1.4. Feature Scaling and Normalization

Scaling values to a specific range (like 0 to 1) or calculating Z-scores (distance from mean in standard deviations) ensures that features with large ranges don't dominate models.

SELECT 
    val,
    (val - MIN(val) OVER()) / (MAX(val) OVER() - MIN(val) OVER()) AS min_max_scaled
FROM measurements;
Output: Returns values scaled between 0 and 1.

VII.1.5. Random Sampling for Training

To reduce data volume and explore large datasets, Data Scientists use sampling. TABLESAMPLE or RANDOM() can create random subsets, but representativeness depends on the sampling method and the data.

Common mistakes: Forgetting to handle NULLs before scaling, which can return NULL results for the entire feature.

Quick recap: Feature engineering in SQL shifts the computational load from Python/R to the database, enabling the processing of larger datasets.

Practice question: How do you calculate a Z-score for a column height?

Answer: (height - AVG(height) OVER()) / STDDEV(height) OVER()

VII.2. Analytics Cookbook (Business Intelligence Patterns)

Learning goals: Apply complex SQL patterns to solve real-world business and data science problems.

Prerequisites: Section I.1., II.3., IV.2., VII.1.

Key terms: Retention, Cohort Analysis, YoY Growth, Cumulative Sum.

VII.2.1. Retention and Cohort Analysis

Cohorts are groups of users sharing a common characteristic (usually their signup month). Analyzing their behavior over time helps identify churn patterns.

-- Get the first signup month for each user
WITH first_order AS (
    SELECT user_id, MIN(DATE_TRUNC('month', order_date)) AS cohort_month
    FROM orders GROUP BY 1
)
SELECT 
    f.cohort_month,
    DATE_TRUNC('month', o.order_date) AS active_month,
    COUNT(DISTINCT o.user_id) AS active_users
FROM orders o
JOIN first_order f ON o.user_id = f.user_id
GROUP BY 1, 2 ORDER BY 1, 2;

VII.2.2. Running Totals and Cumulative Metrics

Running totals track progress over time (e.g., total sales to date). This is achieved using window functions with the default frame (ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW).

SELECT 
    date, 
    daily_revenue,
    SUM(daily_revenue) OVER (ORDER BY date) AS cumulative_revenue
FROM daily_sales;

VII.2.3. Year-over-Year (YoY) Growth

Comparing current performance to the same period in the previous year is a staple of business reporting. LAG() with an offset of 12 (for monthly data) or 1 (for yearly data) is the standard approach.

WITH yearly_sales AS (
    SELECT EXTRACT(year FROM date) AS year, SUM(revenue) AS revenue
    FROM sales GROUP BY 1
)
SELECT 
    year, 
    revenue,
    LAG(revenue) OVER (ORDER BY year) AS prev_year_revenue,
    (revenue - LAG(revenue) OVER (ORDER BY year)) / LAG(revenue) OVER (ORDER BY year) AS growth_pct
FROM yearly_sales;

VII.2.4. Manual Pivot Tables (Long to Wide)

While some databases have a PIVOT command, the most portable way to pivot data is using conditional aggregation (SUM(CASE...)).

SELECT 
    region,
    SUM(CASE WHEN quarter = 1 THEN revenue ELSE 0 END) AS q1_revenue,
    SUM(CASE WHEN quarter = 2 THEN revenue ELSE 0 END) AS q2_revenue
FROM regional_sales
GROUP BY region;

Common mistakes: Using DATE_PART instead of DATE_TRUNC for cohorts (losing the year context), forgetting to handle division by zero in growth calculations.

Quick recap: Use LAG() for growth, SUM() OVER() for running totals, and CASE inside SUM() for pivoting.

Practice question: How do you calculate a 7-day moving average of sales?

Answer: AVG(sales) OVER(ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)
